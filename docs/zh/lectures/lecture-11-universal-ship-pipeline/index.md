研究只有在结果被发布后才算完成。`/autoresearch:ship` 是一个8阶段流水线，通过一个强制性的人工确认关卡，将任何制品从"就绪"带到"已部署"。

8个阶段依次为：**识别（Identify）→ 清单（Inventory）→ 检查清单（Checklist）→ 准备（Prepare）→ 演习（Dry-run）→ 确认（CONFIRM）→ 发布（Ship）→ 验证+记录（Verify+Log）**。其中第六阶段"确认"是**唯一需要人工输入的阶段**：展示清单、检查清单结果和演习输出，要求明确确认后才继续。若所有检查通过且设置了 `--auto`，则自动批准。`/autoresearch:ship` 支持9种发布类型：代码 PR（推送分支、创建含描述的 PR、请求审阅者）、代码发布（升级版本、生成变更日志、打标签、创建 GitHub Release）、部署（构建、推送容器、部署到环境、健康检查）、内容发布、营销邮件、营销活动、销售、研究成果发布、设计资产发布。每种类型生成不同的领域特定检查清单，所有检查项均为机械可验证的——没有主观的"看起来不错"。

发布动作通过 `ship-log.md` 完整记录，`--rollback` 可撤销最后一次发布：代码 PR 删除分支，部署执行 `kubectl rollout undo`，内容发布回滚到上一版本。推荐的完整研究流水线是：`plan → autoresearch → debug → fix → security → ship`。

## 核心概念

**8阶段流水线**：从自动识别发布类型到发布后验证和记录的完整流程，确保无遗漏步骤。

**强制人工确认关卡（第6阶段）**：流水线中唯一需要人工输入的点，在任何不可逆操作之前展示完整信息请求确认。

**9种发布类型**：覆盖代码 PR、版本发布、部署、内容、邮件、营销活动、销售、研究成果、设计资产——同一框架适用于所有类型的制品。

**机械可验证检查清单**：每个检查项都是可自动运行的命令，没有主观判断，确保"就绪"的定义是客观的。

**`--rollback` 可逆性**：每次发布动作都以足够的信息记录在 `ship-log.md` 中，支持完整回滚。

## 关键要点

- 8个阶段从识别到记录提供完整覆盖
- 强制人工确认关卡（第6阶段）是唯一需要人工输入的点
- 9种发布类型覆盖从代码 PR 到营销活动的所有制品
- 检查清单是机械可验证的——没有主观的"看起来不错"
- `--rollback` 使每次发布动作可撤销
- 完整研究流水线链：plan → autoresearch → debug → fix → security → ship
