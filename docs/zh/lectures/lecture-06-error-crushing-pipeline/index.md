`/autoresearch:debug` 负责发现 bug，`/autoresearch:fix` 负责消灭错误。这是两个不同的问题，解决策略也不同。调试是调查：你不知道哪里出了问题。修复是修缮：你已有一份错误清单，目标是将其归零。修复循环针对系统性、级联感知的错误消除进行了专门优化。

修复循环中最核心的挑战是**级联问题**：修复错误 A 可能暴露原本被掩盖的错误 B，修复错误 B 可能破坏原本通过的错误 C。`/autoresearch:fix` 通过级联感知策略应对这一问题：每次迭代只修复一个错误，按优先级排序（构建错误 → 类型错误 → 测试失败 → 代码风格错误），每次修复后验证，并通过 Guard 防止退化。每次迭代的流程是：运行验证命令获取当前错误列表 → 选取最高优先级的未修复错误 → 形成具体修复假设 → 做出单一修改 → `git commit` → 再次运行验证 → 运行 Guard → 保留或回滚。循环在错误计数归零时自动停止，即使在无界模式下也是如此。

最强大的使用方式是 `--from-debug` 链式命令：先运行 `debug` 找出所有 bug，再运行 `fix --from-debug` 读取 `findings.md` 中已确认的根因，跳过发现阶段直接进入修复。三种机制防止回归：Guard（在每次修复后运行完整绿色测试套件）、每次迭代单一改变（回归责任明确），以及自动回滚（Guard 失败且二次修复也失败时，原始改变被撤销并记录）。

## 核心概念

**级联感知排序**：按"构建错误 → 类型错误 → 测试失败 → 代码风格错误"顺序修复，先修复阻塞下游的错误，避免在依赖性错误上浪费迭代。

**每次修复一个错误**：保持回归责任明确——出现问题时始终知道该回滚哪一个提交。

**Guard 防回归**：在每次修复后运行绿色测试套件，确保修复不破坏其他已通过的测试。

**`--from-debug` 链式命令**：读取 `findings.md` 中已确认的根因，将调查发现直接导入修复流程，跳过重新发现阶段。

**不可修复错误处理**：对于单个错误连续3次修复失败的情况，记录为 SKIP 并继续处理下一个，防止无限循环。

## 关键要点

- 修复和调试是不同的问题——修复假设你已知道哪里出了问题
- 级联感知排序（阻塞者优先）防止在依赖性错误上浪费迭代
- 每次迭代一个修复使回归责任明确——你始终知道需要回滚什么
- Guard 防止回归积累——每次修复都对照绿色基准进行验证
- `--from-debug` 将调查发现直接链入修复，跳过重新发现过程
- 循环在零错误时自动停止——无需人工监视
