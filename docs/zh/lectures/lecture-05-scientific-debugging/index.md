大多数调试并不科学，而是"考古式"的：开发者翻查日志、同时改动多个变量、最终碰巧找到修复方案——却从未真正理解为什么有效。`/autoresearch:debug` 将科学方法应用于 bug 调查：每次迭代一个可证伪的假设、系统性地收集证据，并永久记录测试过的内容和所学到的知识。

调试循环分为7步：收集症状 → 侦察（绘制错误范围）→ 形成假设 → 设计并运行单一实验 → 分类（确认/否定/无结论）→ 记录 → 重复。循环在所有假设分类完毕且修复已应用，或 `max_iterations` 耗尽时终止。agent 可从7种调查技术中选择：二分搜索、差异调试（`git bisect`）、最小复现、执行追踪、模式搜索、反向追踪、橡皮鸭调试。agent 会根据 bug 类型自动选择技术，并在3次迭代未能缩小搜索范围时切换技术。

调试会话生成三个输出文件：`hypotheses.md`（所有假设及其状态和证据）、`eliminated.md`（已排除原因的快速参考清单）、`findings.md`（确认的 bug、复现步骤和推荐修复方案）。**`eliminated.md` 是最有价值的输出**——它防止未来的开发者重新调查死路，是可积累的机构知识。使用 `/autoresearch:debug --fix` 可在确认根因后自动切换到修复流程。

## 核心概念

**一次假设原则**：每次迭代只测试一个具体、可证伪的假设，避免多变量改动导致无法归因。

**否定与确认同等有价值**：知道"什么不是原因"是真正的进展，能防止在死路上浪费时间。

**7种调查技术**：二分搜索、差异调试、最小复现、执行追踪、模式搜索、反向追踪、橡皮鸭调试——根据 bug 类型动态选择。

**`eliminated.md`**：已排除假设的干净列表，是跨会话可复用的机构知识。

**`--fix` 链式命令**：在 `debug` 确认根因后自动切换到 `fix` 进行修复，构成完整的调查-修复流水线。

## 关键要点

- 科学调试要求每次迭代一个假设，而非多变量同时修改
- 否定假设与确认假设同等有价值——知道什么不是原因就是进展
- 7种技术并非顺序执行，agent 根据 bug 类型选择并在卡住时切换
- `eliminated.md` 是最有价值的输出——防止重新调查死路
- 使用 `/autoresearch:debug --fix` 将调查发现直接链入修复流程
- `hypotheses.md` 中的永久记录是机构知识——未来的开发者从你停下的地方继续
