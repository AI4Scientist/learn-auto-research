在调试、修复或发布之前，`/autoresearch:predict` 在两分钟内提供五个专家视角的分析。这是一种"飞行前检查"：在采取行动前先理解风险全貌。

大多数预先分析都受到**视角单一化**的困扰：一个人（或一个 AI 角色）从其习惯角度评估变更——安全工程师看安全风险，性能工程师看延迟风险，可靠性工程师担心边界情况，但他们很少同时审视同一变更。`/autoresearch:predict` 通过运行五个独立的专家视角分析来解决这个问题：**架构师**（系统设计、耦合、未来可扩展性）、**安全分析师**（威胁建模、攻击面、信任边界）、**性能工程师**（计算复杂度、内存分配、高负载行为）、**可靠性工程师**（故障模式、优雅降级、爆炸半径）、**魔鬼代言人**（反驳最可能错误的假设、提出更简单的替代方案）。

防止"群体思维"是本讲最重要的设计原则：每个角色在**冷启动独立分析**后才进入合成阶段，不会看到其他角色的输出。合成阶段明确寻找分歧：4/5同意的发现（高置信度）、仅1/5提出的发现（少数意见——最有价值，往往捕捉到共识遗漏的信息）、两个角色直接矛盾之处（未解决的张力——标记供人工决策）。输出文件包含共识发现、值得关注的少数意见和未解决张力，配合 `--chain debug/security/fix` 可将发现直接导入行动命令。

## 核心概念

**视角单一化**：单一角度评估带来的盲点——安全专家看不到性能问题，性能专家忽略安全风险。

**冷启动独立分析**：每个专家角色在不看其他角色输出的情况下独立分析，防止群体思维和锚定效应。

**少数意见（minority view）**：仅1-2个角色提出的发现，往往是整个分析中最有价值的输出，捕捉共识遗漏的信息。

**未解决张力**：两个专家角色直接矛盾的地方，需要人工做出判断。

**`--chain` 链式命令**：将五专家分析的发现直接导入 `debug`、`security` 或 `fix` 命令，生成预排序的调查假设列表。

## 关键要点

- 五个独立冷启动分析防止向有缺陷的共识收敛
- 每个角色有专属分析视角——无重叠、无盲区
- 少数意见是最有价值的输出——它们捕捉共识遗漏的信息
- 未解决的张力需要人工决策——明确标记，不掩盖
- 通过 `--chain debug/security/fix` 将发现直接导入行动命令
- 最适合使用 predict 的场景：大型重构前、合并大型 PR 前、生产部署前
